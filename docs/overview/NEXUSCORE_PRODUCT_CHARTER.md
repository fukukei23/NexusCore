# NexusCore Product Charter

## 1. 存在意義（WHY）— なぜNexusCoreが存在するのか

### 1.1 問題の根源

ソフトウェア開発の歴史は、**「同じ過ちを繰り返し、組織知識が個人に閉じ込められる」**という痛恨のループである。コードは書かれるが、学習は蓄積されない。優秀なエンジニアが退職すれば、その組織の知的資産も共に消える。

従来のAIアシスタントは、個人の生産性を上げることはできても、**組織全体の学習能力を高めることはできなかった**。個人がプロンプトを出し、AIが回答し、それが終わる。経験は残らない。同じミスを別のエンジニアが別の日に繰り返す。

### 1.2 我々が信じる原則

NexusCoreは、以下の信念に基づいて構築されている。

**組織とは、個人の貢献者の集合ではなく、共有された知識体の集合であるべきだ。**

我々は、AIエージェントを「コードを生成する道具」ではなく、「**組織の学習器官**」として設計する。NexusCoreが存在する理由はただ一つ：

> **「組織が一度犯した失敗を、二度と繰り返さない世界の実現」**

これは単なるスローガンではない。NPE（Nexus Protocol Engine）が予算超過を防ぐように、FKB（故障知識ベース）が過去のパターンを記憶するように、すべての設計判断は「知識の蓄積と再利用」という目的関数に従っている。

### 1.3 技術への思想

多くのAI開発ツールは「速さ」を殊更に喧伝する。しかし、我々は敢えてこう問う：**速いが再現できないことと、遅いが再現できることの、どちらが組織にとって価値があるか？**

答えは明白だ。エンタープライズにおいて、監査対応の証明可能性は単なるコンプライアンス要求ではない。それは組織の生存能力そのものである。NexusCoreは、証跡（Audit Trail）を副次的成果としてではなく、**第一級の設計要件**として位置づける。

## 2. プロダクト定義 — NexusCoreとは何か、何でないか

### 2.1 NexusCoreとは何か

NexusCoreは、**エンタープライズ向けの自律型マルチエージェントAI開発プラットフォーム**である。

18以上の専門エージェント（要件解析、設計、計画、コーディング、テスト、デバッグ、監視、ポストモーテム等）が、Orchestratorを中心に協調動作し、複雑なソフトウェア開発タスクを End-to-End で実行する。

### 2.2 NexusCoreの構成要素

| コンポーネント | 役割 |
|---|---|
| **Orchestrator** | タスクの分解、工数の見積、エージェント間の調整 |
| **Specialized Agents** | 各専門領域（要件・設計・実装・テスト等）を担う18以上のエージェント |
| **FKB (Fault Knowledge Base)** | 失敗パターンの蓄積・学習・再利用 |
| **NPE (Nexus Protocol Engine)** | セキュリティ、予算、機密情報管理の統制 |
| **Quality Gates** | 静的解析（Tier 1）+ ミューテーションテスト（Tier 2）による品質保証 |
| **AuthorityLevel System** | HUMAN_CONTROLLED → PARTIALLY_AUTONOMOUS → FULLY_AUTONOMOUS の段階的自律性 |

### 2.3 NexusCoreとは何でないか

- **個人の生産性向上ツールではない**: 個人利用よりも、組織全体の知的資産蓄積を主目的とする
- **単純なコード生成AIではない**: 14のエージェントが協調し、自己修復・自己進化する統合システム
- **ベータテスト中の実験的プロダクトではない**: 4,895のテストケースと80%以上のコードカバレッジを持つ本番稼働可能なシステム
- **汎用AIアシスタントのEnterprise版ではない**: ガバナンス・監査性・コード機密性を最優先に設計された特化型プラットフォーム

## 3. コアバリュー — 競合優位性と差別化ポイント

### 3.1 自己進化する開発エコシステム

既存のAIコーディングツール（GitHub Copilot等）が「**個人の能力最大化**」を追求するのに対し、NexusCoreは「**組織の知識としての蓄積**」を追求する。

- **Copilot**: あるエンジニアが同じミスをした場合、その知識は次のプロジェクトには引き継がれない
- **NexusCore**: あるエージェントが失敗を記録すれば、同じ失敗は二度と発生しない

これが「組織の七転び八起き」ではなく「**七転び一起き（で十分）**」を可能にする。

### 3.2 エンタープライズ・ガバナンスへの特化

| 特性 | 汎用AIアシスタント | NexusCore |
|---|---|---|
| コード機密性 | クラウド送信・学習の懸念 | NPEによるマスキング・統制 |
| ガバナンス | なし | Constitutionによる品質基準定義 |
| 監査性 | 限定的 | 完全な証跡・再現性 |
| 自律性の段階的制御 | なし | AuthorityLevelによる制御 |
| 予算超過防止 | なし | NPEによるリアルタイム管理 |

NexusCoreは、「便利さ」よりも「**制御可能性と信頼性**」を選ぶ組織のためのプラットフォームである。

### 3.3 Defense in Depth（多層防御）

NPE（Nexus Protocol Engine）は、Zero Trust原則とDefense in Depthアーキテクチャに基づき、以下の多層防御を提供する：

1. **入力検証**: 要件、プロンプト、エンドポイントのリクエスト検証
2. **権限管理**: 最小権限の原則に基づくアクセス制御
3. **予算統制**: リアルタイムコスト監視・超過防止
4. **機密情報マスキング**: コード中の機密情報を自動検出・保護
5. **監査証跡**: すべての操作の完全記録

## 4. ターゲットとビジネスモデル

### 4.1 ターゲット顧客

| セグメント | 特性 |
|---|---|
| **Enterprise（大企業）** | セキュリティ要件が厳しく、コード資産の社外流出を極度に警戒する組織 |
| **Regulated Industries** | 金融、医療、官公庁など、監査対応が法規制で求められる業種 |
| **Tech Companies（大規模開発組織）** | 数百人規模のエンジニア組織で、知識共有・品質標準化が必要な企業 |

### 4.2 顧客ニーズの理解

これらの顧客がNexusCoreに求めるのは、「AIにコードを書かせること」ではなく、「**AIに正しく・安全に・記録を残してコードを書かせること**」である。

彼らにとって、開発速度の10%向上より、コード機密性の99%保証の方が遥かに重要である。

### 4.3 ビジネスモデル

SaaS（Software as a Service）モデルを採用する。

- **GitHub OAuth** によるSSO統合
- **Celery非同期処理** によるスケーラブルなタスク実行
- **PostgreSQL** によるデータ永続化
- **Docker/Kubernetes** によるコンテナ化されたデプロイ

## 5. 設計原則（Design Principles）— 技術判断の指針

NexusCoreのすべての技術的判断は、以下の原則に基づいて行われる。

### 5.1 自己正当化の禁止 vs. 自己修復の許可

**禁止：自己正当化**
根拠なく「これは正しい」と主張するAIの判断は、エンタープライズ環境では受け入れられない。NexusCoreのエージェントは、すべての判断に根拠（reasoning）を明示する必要がある。

**許可：自己修復**
失敗を認め、FKBを更新し、次回から改善することは、むしろ**奨励される**。失敗を隠蔽する文化よりも、失敗から学習する文化が組織の資産となる。

### 5.2 不可逆変更の禁止 vs. 進化の許可

**禁止：不可逆変更**
本番環境のコードに対して、ロールバックできない変更を自動的に適用することは、エンタープライズにおいては許容されない。すべての変更は段階的に、検証可能に行われる。

**許可：進化（試行錯誤）**
実験的に新機能を試みることは、テスト環境内で許可される。失敗した実験は知識として蓄積され、組織全体の学習となる。

### 5.3 AIは提案者であり、最終決定者ではない

NexusCoreのエージェントは、最終的な判断を **Human** に委ねることを基本とする。AuthorityLevelがFULLY_AUTONOMOUSに設定された場合であっても、責任所在の明確化と監査可能性の確保は変わらない。

### 5.4 速度より安全性・証跡・再現性を優先

「早く終わらせる」ことより「正しく終わったことを証明できる」ことが、エンタープライズにおいては遥かに価値が高い。

```
優先順位：
1. 安全性（Security & Safety）
2. 証跡（Audit Trail）
3. 再現性（Reproducibility）
4. 速度（Speed）
```

### 5.5 失敗は負債ではなく資産

従来のソフトウェア開発では、バグや障害は「技術的負債」と見なされた。しかしNexusCoreでは、適切にドキュメント化された失敗は、**組織の最も価値ある知的資産**である。

FKBに記録された失敗パターンは、未来の開発を加速させる教材であり、新人エンジニアの最短習得パスであり、業界全体のベストプラクティス共有の基盤である。

### 5.6 品質はチェックではなく、統合されたゲートである

品質は「最後の砦」ではなく、**開発の各段階で統合されたGate**である。Tier 1（静的解析）とTier 2（ミューテーションテスト）は独立した防衛層であり、Constitution（憲法）が品質基準を定義する。

## 6. 成功指標（Success Metrics）

### 6.1 プロダクト的成功指標

| 指標 | 測定方法 | 目標 |
|---|---|---|
| **FKB利用率** | エージェントがFKBを参照した回数 / 総実行回数 | 上昇傾向を維持 |
| **自己修復率** | エージェントが自律的にエラーを修正した割合 | 80%以上 |
| **品質ゲート通過率** | 最初のコミットで品質ゲートを通過する割合 | 70%以上 |
| **設計原則遵守率** | 各設計原則に違反した判断の割合 | 5%未満 |

### 6.2 ビジネス的成功指標

| 指標 | 測定方法 | 目標 |
|---|---|---|
| **顧客維持率** | 年間継続契約率 | 90%以上 |
| **NPS** | 顧客満足度調査 | 50以上 |
| **導入企業数** | 有償顧客企業数 | 四半期ごとに15%成長 |
| **エンタープライズ比率** | 大企業顧客 / 全顧客 | 60%以上 |

### 6.3 技術的健康性指標

| 指標 | 測定方法 | 目標 |
|---|---|---|
| **テストカバレッジ** | 自動テストのコードカバレッジ | 80%以上（現状80.22%） |
| **テストケース数** | 実行可能テストケース総数 | 5,000以上 |
| **稼働率** | サービス可用性 | 99.9%以上 |
| **インシデント解決時間** | 重大障害からの平均回復時間 | 30分以内 |
