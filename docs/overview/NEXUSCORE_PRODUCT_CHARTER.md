# NexusCore Product Charter（軽量版）

## 1. Vision（存在理由）

NexusCore が解こうとしている本質的な問題は、「ソフトウェア開発が *人間の頭の中* に閉じたまま、失敗と学びが再利用されにくい」ことにある。
NexusCore は単なるコード生成ではなく、要求→計画→設計→実装→テスト→レビューという **開発プロセスそのもの**を、再現可能な手順と証跡の上で実行できるようにする基盤である。
AI には実行・提案・探索を任せるが、最終的な責任（承認・例外判断・公開判断）は人間が持つ。これは速度よりも **安全性・証跡・再現性**を優先するための立場である。

## 2. Non-Goals（やらないこと）

- 完全ブラックボックスの自律（説明できない判断）を目指さない
- 人間責任を AI に転嫁しない（「AI がそう言ったから」は根拠にならない）
- 巨大な単一の司令塔に意思決定を詰め込まない（役割を分ける）
- 説明できない最適化を採用しない（監査不能な shortcut を許容しない）

## 3. Core Principles（設計原則）

### Policy over Engine
動作の正当性はエンジンの賢さではなく、明示されたポリシーと境界で担保する。
「何をしてよいか/いけないか」を先に固定し、実行系はその範囲で最適化する。
ルールは暗黙ではなく文書と証跡で残す。

### Freeze the Core
破壊的影響が大きい中核は、まず壊さない前提で守る。
変えるなら小さく、段階的に、根拠とロールバック込みで進める。
外側の薄い層で制御できる余地を残すことが、進化の安全弁になる。

### Human-in-the-loop by Design
人間介入は「失敗時の例外」ではなく、設計上の前提として配置する。
承認・停止・例外判断のポイントを最小単位で切り出し、運用に組み込む。
AI は提案者であり、最終決定者ではない。

### Audit-first
説明責任は追加機能ではなく、最初から設計の中心に置く。
いつ/何を/なぜ/どこまで行ったかを追えることが、安全と再現性を生む。
成功より失敗の証跡を重視する（再発防止を資産化する）。

### Spec-driven Development
要求（SRS）→実装仕様（CR）→検証（テスト/レポート）の順で前進する。
仕様の空白を「勢い」で埋めず、判断の根拠を文書に残す。
変更は最小差分で行い、後から追跡可能にする。

## 4. System Model（全体構図）

NexusCore は「実行系」と「判断/責任」と「監査」を分離して成立する。

- **Human**: 統治者・承認者。例外判断、公開判断、最終責任を担う。
- **Orchestrator**: 実行の統制者。フェーズ順序・境界・停止判断を強制する。
- **Agents**: 実働部隊。収集、解析、提案、修復、検証などの専門作業を担う。
- **Policy / Governance**: 禁止事項・凍結境界・説明責任のルールを固定する。
- **Observability / Evidence**: ログ・差分・失敗分析などの証跡を保持する。

Orchestrator は「実行エンジン」であり、意思決定者ではない。意思決定はルール（Governance）と人間の承認に従属する。
AuthorityLevel は **権限境界**であり、命令系（“何を作るか”）ではない。許容される自律範囲を定義し、越境を防ぐための設計要素である。

## 5. Authority & Control Philosophy

AuthorityLevel が必要なのは、AI の能力が上がるほど「速く間違える」リスクも上がるからである。
自律性を段階化し、どこまでを自動で進め、どこで人間介入を要求するかを **事前に合意可能**にすることで、運用と監査が成立する。

中核を凍結し、外側で制御するのは「進化を止める」ためではなく、「壊す順番を守る」ためである。
最小安全単位としてフェーズ境界制御から始めるのは、影響範囲が明確で、停止・ロールバック・説明がしやすいからである。
この思想は SRS の要求（AuthorityLevel/互換/導線/テスト可能性）と、関連 CR の実装仕様に整合する。

## 6. Governance & Accountability

Charter / Governance / SRS は役割が異なる。

- **Charter**: 思想・構図・判断原則（なぜ/どう考えるか）
- **Governance**: 制約・責任・禁止事項（何をしてはいけないか/誰が責任を持つか）
- **SRS**: 要求仕様（何を満たすべきか：FR/NFR）

監査・説明責任を第一級に置く理由は、NexusCore の価値が「成果物」だけでなく「プロセスの信頼性」にあるためである。
外部提供（SaaS/API）に耐えるには、速度よりも証跡・再現性・責任境界が優先される。

## 7. Evolution Model（進化前提）

現在は v0.x として、外側制御・フェーズ単位の安全境界を重視する段階である。
将来、より大きな変更（中核を壊す必要がある変更）に着手する局面が来る。その際は、先に要求を固定し、設計を整え、実装へ落とす順番を守る。

- 進め方の原則: **SRS → SDD → CR**
- 進化は「不可逆」を避け、段階的に、根拠とロールバック込みで進める

※ロードマップの詳細は `docs/overview/03_Development_Roadmap.md` を参照する（本 Charter では列挙しない）。

## 8. Document Roles & Traceability

| 文書 | 役割 |
|---|---|
| Product Charter | 思想・構図・判断原則 |
| SRS | 要求仕様（FR/NFR） |
| CR Specs | 実装仕様（HOW） |
| Governance | 制約・責任 |
| Overview Docs | ビジネス/構想/運用の補助説明 |

トレーサビリティは「要求（SRS）を満たす実装（CR）」として一方向に保つ。
Charter は要求や実装を新規に定義せず、判断の拠り所（なぜ/どう守るか）を与える。
