## 1. Purpose（このプロファイルの目的）
本書は NexusCore の開発・運用における「制約（Constraints）」を宣言し、AI実装・レビュー・意思決定が推測で逸脱しないよう統制する。

**本書の位置づけ**: 本書は NexusCore プロジェクト制約の正本（SSOT: Single Source of Truth）である。開発タスク開始前に必ず参照し、本書に記載された制約に従うこと。

- 何を固定するか：
  - スコープ境界（やる/やらない）
  - 非機能要件（品質・安全・運用）
  - 禁止事項（やってはいけない）
  - 変更管理（Spec / Phase 3 / Decision Log）

## 3. Operating Model（運用モデル）
### 3.1 Gate（参照強制）
開発タスク開始前に必ず参照する：
- `docs/ARCHITECTURE.md`（SSOT入口）
- [Project Profile](../プロジェクトプロファイル/PROJECT_PROFILE_NEXUSCORE.md)（このファイル）
- [Governance](../ガバナンス/README.md)
- `ガバナンス/マスタープロトコルテンプレート.md`（プロトコル）
- 変更対象に関連する Spec（`docs/spec/` または `ガバナンス/spec/`）

### 3.2 STIT（Spec & Test Driven Iteration）
- 仕様（Spec）を先に書く。テスト可能な受入条件を含める。
- 実装は Spec を満たす最小差分に限定する。

### 3.3 IRG（Independent Review Gate / Phase 3）
- 実装者とは別コンテキスト（別チャット/別AI/別セッション）でレビューする。
- IDE内の自己チェックは Gate 通過とみなさない。
- 結果は Decision Log に必ず転記する（Reject含む）。

## 4. Quality & Non-Functional Requirements（非機能要件）
### 4.1 Reliability
- 失敗時に復旧可能（ログ・差分・再実行手順が残る）
- 同一入力で再現可能な振る舞い（可能な範囲で）

### 4.2 Security / Secrets
- APIキー、トークン、個人情報を Git 管理下に入れない
- ログ/ZIP出力/共有物に秘密情報が混入しない設計を優先
- 外部LLMへ送信する情報は最小化し、機密は原則遮断

### 4.3 Observability（観測性）
- 重要イベント（実行、失敗、修正、レビュー結果）を追跡可能にする
- 何がいつ変わったかを Git と Decision Log で追える状態を維持

## 5. Constraints（制約）
### 5.1 Architectural Constraints
- SSOTはドキュメント（ARCHITECTURE/Spec/Decision Log）に置く。UIに依存させない
- 責務境界：
  - Agent：判断/生成
  - Utility：I/O、ファイル操作、差分適用、テスト実行
  - Router：モデル選択/予算/ポリシー
- 既存構造の大規模移動は、別Specで段階的に扱う（本プロファイルでは抑制）

### 5.2 Entry Points（入口①②③の責務境界）
NexusCore の主要な入口コンポーネントとその責務境界を定義する。

#### 入口①: NexusGuard（品質ゲート・セキュリティチェック）
- **責務**: コード品質、セキュリティ、プロジェクト憲法への準拠をレビュー
- **境界**:
  - コードレビュー・承認プロセスの最終決定権を持つ
  - Git コミット前の自動セキュリティスキャンを実施
  - 不正コードの承認・コミットを防止
- **禁止事項**:
  - 承認ログの改ざん
  - マルチ承認フローなしでの自動コミット
  - セキュリティスキャン未実施での承認

#### 入口②: NexusEval（評価・テスト実行）
- **責務**: テスト生成・実行・結果の可視化
- **境界**:
  - テスト実行環境の管理（pytest等）
  - テスト結果の収集・集計
  - 評価メトリクスの算出
- **最小前提**:
  - **task_type = JSON構造出力**: 評価タスクは JSON 構造で出力されることを前提とする
  - **正解 = 制約条件集合（教師データ非前提）**: 正解は教師データではなく、制約条件集合として定義される
- **禁止事項**:
  - 教師データへの依存
  - 非構造化出力の前提

#### 入口③: NexusTrace（トレーシング・ログ記録）
- **責務**: 重要イベント（実行、失敗、修正、レビュー結果）の追跡
- **境界**:
  - ログ・差分・再実行手順の記録
  - Git と Decision Log による変更履歴の追跡
  - 観測性の確保
- **禁止事項**:
  - ログの改ざん・削除
  - 秘密情報のログ混入
  - トレース情報の欠落

### 5.3 Documentation Constraints
- **ガバナンス資産（ガバナンス/**, プロジェクトプロファイル/**, 判断ログ/**）内に絶対パスや環境依存パスを書かない**（例示でも禁止）
- `docs/` 配下の既存ドキュメントは対象外（WSL環境での運用手順として必要なパスを含む場合がある）
- チャット由来の疑似パス/疑似アーティファクト文字列を「実在」として扱わない
- SSOT（Git管理ファイル）以外を根拠にしない
- 衛生チェック対象は **ガバナンス/**, **プロジェクトプロファイル/**, **判断ログ/** に限定する（`docs/` は対象外）

## 6. Must Not（禁止事項：最優先事故）
以下の事故は **最優先で防止** しなければならない。これらが発生した場合、システムの信頼性・安全性が根本的に損なわれる。

### 6.1 Tier-0: 最優先事故（候補A採用）
NexusEval の判定基準に直結する、以下の3項目は **絶対に発生させてはならない**。

#### 6.1.1 誤った AI 判断結果が、人の確認なしに本番業務へ反映されること
- **事故**: AI の判断結果が自動的に本番環境に適用され、人的確認を経由しない
- **影響**: 誤った判断による本番障害、ビジネス損失、信頼性の根本的損傷
- **防止策**:
  - 本番適用前の人的承認フローの必須化
  - 自動適用の禁止（Staging/Preview 環境での検証は可）
  - 判断結果の可視化と承認待ち状態の明示

#### 6.1.2 評価結果の根拠（入力・出力・判定理由）が後から再現・説明できないこと
- **事故**: 評価結果の根拠となる入力データ、出力データ、判定理由が記録されていない、または再現不可能
- **影響**: 評価結果の検証不可、意思決定の根拠不明、監査対応不可
- **防止策**:
  - 評価実行時の入力・出力・判定理由の完全記録（不変ログ）
  - 評価結果と根拠データの紐付け（correlation_id 等）
  - 再現可能な評価環境の維持（入力データの保存、実行環境の記録）

#### 6.1.3 AI 評価の数値が意思決定に使えない曖昧な形で提示されること
- **事故**: 評価結果が数値として提示されるが、その意味・閾値・信頼区間が不明確で意思決定に使えない
- **影響**: 意思決定の根拠不明、評価結果の誤解釈、適切な判断ができない
- **防止策**:
  - 評価数値の意味・単位・範囲の明示
  - 閾値・判定基準の事前定義と明示
  - 信頼区間・不確実性の提示（可能な範囲で）

### 6.2 Tier-1: セキュリティ・運用上の重大リスク
以下の事故は、セキュリティ・運用上の重大リスクとして防止しなければならない。

#### 6.2.1 認証・認可の欠如
- **事故**: APIエンドポイントが認証なしで公開される
- **影響**: 不正アクセス、データ漏洩、サービス停止攻撃
- **防止策**:
  - APIキー認証またはOAuth2.0/JWT認証の実装
  - レート制限（Rate Limiting）の導入
  - IPホワイトリスト/ブラックリスト機能

#### 6.2.2 不正コードの承認・コミット
- **事故**: GuardianAgent が不正コードを承認・コミットする
- **影響**: Git履歴の改ざん、セキュリティ脆弱性の混入
- **防止策**:
  - マルチ承認フロー（複数レビュアーの承認が必要）
  - 承認ログの不変性保証（署名付きログ）
  - コミット前の自動セキュリティスキャン

#### 6.2.3 機密情報のLLM送信
- **事故**: 機密情報が検出漏れにより外部LLMへ送信される
- **影響**: 機密情報の漏洩、コンプライアンス違反
- **防止策**:
  - 機密情報検出・マスキングの強化（正規表現ベース以外の手法も検討）
  - 外部LLMへ送信する情報の最小化
  - 送信前の機密情報スキャン

#### 6.2.4 Orchestrator フェーズ構造の破壊
- **事故**: Requirement → Plan → Architecture → Coding → Testing → Review のフェーズ構造が変更される
- **影響**: システムの動作保証が失われる、既存エージェントとの互換性が破壊される
- **防止策**:
  - Orchestrator の public メソッドのシグネチャ変更禁止
  - フェーズ構造の変更は別Specで段階的に扱う

#### 6.2.5 diff 適用ロジックの破壊
- **事故**: PatchApplier の diff 適用ロジックが壊される
- **影響**: 自己修復機能が動作しなくなる、システムの自律性が失われる
- **防止策**:
  - diff 適用ロジックは最重要領域として保護
  - 例外握りつぶし禁止
  - 副作用削除禁止（パッチ結果はログ化必須）
