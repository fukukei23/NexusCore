## レビュー観点

### 1. セキュリティリスク

- SQLインジェクション
- XSS脆弱性
- 認証・認可の不備
- 機密情報のハードコード
- 不適切なエラーハンドリング

### 2. パフォーマンスボトルネック

- N+1 クエリ
- 不要なデータフェッチ
- 効率的でないアルゴリズム
- インデックス未使用の検索

### 3. 保守性の問題

- 重複コード
- 適切でない命名
- 複雑すぎる関数
- 魔法の数字や文字列
- コメント不足の複雑なロジック

### 4. NexusCore 規約からの逸脱

- FastAPI を使っていない
- /api/v1 プレフィックスを守っていない
- Pydantic BaseModel を使っていない
- エラーコードカタログ未使用

## レビュー手順

1. 対象ファイルを特定（git log, git diffなど）
2. 各ファイルを順次レビュー
3. 重大な問題、軽微な問題、改善提案に分類
4. 具体的な修正案を提示

## 出力形式

```
## レビュー結果

### 重大な問題
[即座に修正が必要な問題]

### 軽微な問題
[Issue化して後で対応できる問題]

### 改善提案
[コードの品質向上のための提案]

### 全体評価
[全体的な評価と次のステップ]
```

## 注意点

- レビュー結果は適切に分類し、ユーザーが次に取るべきアクションを明確にする
- 重要な指摘がある場合、即座に修正の提案をする
- 決定記録が必要な設計判断が含まれる場合、Decision Recorder エージェントの使用を提案する

## NexusCore 特有のチェック項目
---

- API エンドポイントが `/api/v1/*` になっているか
- 認証が必要なエンドポイントに `Depends()` が使われているか
- エラーレスポンスが標準フォーマットになっているか
- エラーコードが `docs/api/エラーコードカタログ.md` に定義されているか
- Pydantic BaseModel が使われているか
