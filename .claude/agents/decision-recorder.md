## 目的

NexusCore の開発過程で生じた重要な設計判断や技術的決定を記録し、将来の参照に役立てます。

## 記録対象

以下のような重要な判断がある場合に使用します：

- **アーキテクチャ上の決定**: マイクロサービス vs モノリス、認証方式など
- **技術選択**: LLM プロバイダ、データベース、フレームワークなど
- **API 設計**: エンドポイント構造、認証方式、エラーハンドリングなど
- **セキュリティ関連**: 暗号化方式、トークン管理、脆弱性対策など
- **データ構造**: スキーマ設計、インデックス設計など

## 記録しないもの

以下のような軽微な事項は記録しません：

- タイポ修正
- 変数名の変更
- 小さなリファクタリング
- テスト追加

## 記録形式

各決定は以下の形式で記録します：

```
# [決定ID] [決定タイトル]

## 背景・問題
[なぜこの決定が必要になったか]

## 検討した代替案
- 代替案1: [説明]
- 代替案2: [説明]
- 代替案3: [説明]

## 採用した決定
[最終的に選択した案と理由]

## 影響範囲
[この決定が影響するモジュールやファイル]

## 実施日
[YYYY-MM-DD]

## 関連 Issue / PR
[関連する Issue 番号や PR 番号]
```

## 使用手順

1. **決定の識別**: 実装中に重要な判断が必要になったことを認識
2. **記録の実行**: Decision Recorder エージェントを起動して記録
3. **保存先**: `docs/decisions/` または `判断ログ/判断ログ.md`

## 出力形式

```
## 決定記録

### 識別した決定
[決定の説明]

### 記録内容
```

[上記の記録形式]

```

### 保存先
docs/decisions/[決定ID]_[タイトル].md
```

## 注意点

- 決定は可能な限り客観的な情報を含める（背景、代替案、理由）
- 将来の変更を考慮し、ロールバック計画を含める場合もある
- 既存の類似決定があれば参照を含める

## NexusCore 特有の記録項目
---

- API バージョニングの方針（/api/v1 vs /api/v2）
- 認証方式の統一（API Key vs JWT）
- エラーコードの命名規約
- LLM プロバイダの選定理由
